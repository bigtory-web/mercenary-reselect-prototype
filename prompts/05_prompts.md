# 소환 재선택 시스템 — 요청 템플릿 모음

이 파일은 작업 요청문을 복붙해서 쓰기 위한 템플릿 모음이다.  
작업 유형에 맞는 섹션을 골라 `[ ]` 부분만 바꿔 사용한다.

## 공통 사용법

- 제출용 내용 기준은 `01_planning.md`다.
- 구현 상세 확인이 필요하면 `02_dev-spec.md`를 함께 붙인다.
- 수치나 효율 해석이 필요하면 `03_analysis.md`를 함께 붙인다.
- UI 수정 요청은 `prototype/index.html`과 `04_ui-rules.md`를 함께 붙인다.
- 요청문에는 `무엇을 바꿀지`, `무엇은 유지할지`, `결과 형식`을 같이 적는다.

## 1. UI 수정

### 첨부 파일
- `prototype/index.html`
- `04_ui-rules.md`

### 요청 규칙
- 먼저 현재 UI 문제를 3개 이하로 짚어줘.
- 내가 요청한 범위 안에서만 수정해줘.
- 새 화면, 새 패널, 새 플로우는 임의로 추가하지 말아줘.
- 16:9 기준과 내부 스크롤 규칙을 유지해줘.
- 코드 구조가 아니라 실제 화면 결과 기준으로 수정해줘.

### 결과 형식
- 수정 포인트 요약
- 실제 화면이 어떻게 달라졌는지 설명
- 필요하면 확인 포인트 제시

```text
이 HTML 파일을 기준으로 UI를 수정해줘.

첨부 파일:
- prototype/index.html
- 04_ui-rules.md

요청 내용:
[여기에 수정 요청 작성]

유지할 것:
[유지해야 할 요소 작성]

결과 형식:
- 수정 포인트 요약
- 화면 변화 설명
```

## 2. 문서 교정

### 첨부 파일
- `[검토할 파일명].md`

### 요청 규칙
- 내용 자체를 바꾸지 말고 문장과 표현만 다듬어줘.
- 용어를 통일해줘.
- 같은 의미 반복은 줄여줘.
- 제출용이면 더 짧고 단정하게 맞춰줘.

### 결과 형식
- 어색한 표현
- 수정 제안
- 전체 톤 요약

```text
이 문서를 제출용 문장 기준으로 교정해줘.

첨부 파일:
- [검토할 파일명].md

요청 범위:
- 문장 다듬기
- 용어 통일
- 반복 표현 정리

유지할 것:
[내용/구조 중 건드리면 안 되는 것 작성]
```

## 3. PPTX 생성

### 첨부 파일
- `01_planning.md`
- `[참고 PPTX 파일]`

### 요청 규칙
- 참고 PPTX의 톤과 레이아웃을 가능한 범위에서 따른다.
- 16:9 슬라이드 기준으로 구성한다.
- 먼저 슬라이드 목차를 보여주고, 그다음 제작에 들어간다.
- 제출용 문서라면 과도한 요약 슬라이드나 분석 슬라이드는 임의로 추가하지 않는다.

### 결과 형식
- 제안 슬라이드 구성
- 반영 기준
- 제작 결과 파일 경로

```text
이 기획서를 기준으로 PPTX를 만들어줘.

첨부 파일:
- 01_planning.md
- [참고 PPTX 파일]

요청 사항:
- 참고 PPTX의 레이아웃 톤을 따라줘.
- 16:9 기준으로 만들어줘.
- 먼저 슬라이드 구성안을 보여준 뒤 제작해줘.
- 제출용 기준으로 불필요한 슬라이드는 임의로 늘리지 말아줘.
```

## 4. 기획 보완 검토

### 첨부 파일
- `01_planning.md`
- `02_dev-spec.md`

### 요청 규칙
- 개발자가 바로 구현 가능한 수준인지 봐줘.
- 상태 전이, 화면 흐름, 예외 처리, 데이터 규칙의 빈 부분을 찾아줘.
- 중복 설명보다 빠진 설명을 우선 짚어줘.

### 결과 형식
- 부족한 항목
- 왜 부족한지
- 어느 문서에 보완해야 하는지

```text
이 기획 문서를 개발 전달 기준으로 검토해줘.

첨부 파일:
- 01_planning.md
- 02_dev-spec.md

검토 기준:
- 개발자가 추가 질문 없이 구현 가능한지
- 상태 전이와 예외 처리가 충분한지
- 문서 간 역할이 겹치지 않는지

결과는 부족한 점 위주로 정리해줘.
```

## 5. 새 섹션 초안

### 첨부 파일
- `01_planning.md`

### 요청 규칙
- 기존 문체와 섹션 밀도에 맞춰 작성해줘.
- 새 섹션이 전체 흐름에서 어디에 들어가야 하는지도 함께 제안해줘.
- 불필요한 확장 설명은 넣지 말아줘.

### 결과 형식
- 들어갈 위치
- 섹션 제목
- 본문 초안

```text
기존 기획서 문체에 맞춰 새 섹션 초안을 작성해줘.

첨부 파일:
- 01_planning.md

추가할 주제:
- [새로 넣고 싶은 주제]

요청 사항:
- 기존 톤과 문체를 따라줘.
- 섹션이 들어갈 위치도 같이 제안해줘.
- 길이는 현재 문서 밀도에 맞춰줘.
```

## 6. 프로토타입 개발 요청

### 첨부 파일
- `prototype/index.html`
- `02_dev-spec.md`
- `04_ui-rules.md`

### 요청 규칙
- 목적은 완성 코드보다 UX 흐름 검증이다.
- 시스템 규칙은 바꾸지 말고, 레이아웃과 체감을 우선 조정한다.
- 리스트형 구조, 슬롯 오픈형 전환, 우측 액션 영역 유지 원칙을 따른다.
- 핵심 루프는 가능하면 한 화면 안에서 끝나게 유지한다.

### 결과 형식
- 현재 구조 이해
- 수정 포인트
- 구현 결과와 확인 포인트

```text
이 프로젝트의 프로토타입을 UX 검증 기준으로 수정해줘.

첨부 파일:
- prototype/index.html
- 02_dev-spec.md
- 04_ui-rules.md

목표:
- 재선택 UX가 직관적인지 확인하고 싶어.
- 규칙보다 화면 흐름과 체감을 먼저 다듬고 싶어.

수정 원칙:
- 시스템 규칙은 바꾸지 말아줘.
- 리스트형 비교 구조를 유지해줘.
- 슬롯이 열리고 결과가 채워지는 방식으로 보여줘.
- 재선택 비용과 최종 선택 버튼은 액션 영역으로 분리해줘.
- 핵심 루프는 한 화면 안에서 끝나게 유지해줘.
```
