# 소환 재선택 시스템

## 1. 시스템 개요

소환 재선택 시스템은 10회 소환 결과를 확인한 뒤, 중복된 SSR 결과를 후속 선택 기회로 전환하는 보정 시스템이다.

이 시스템의 핵심 목적은 수집 후반부에 반복적으로 발생하는 중복 SSR의 손실 체감을 완화하고, 운 요소로 인해 무너지는 과금 효율 체감을 완충하는 것이다.

## 2. 문제 정의 및 기대 효과

가챠형 수집 구조는 보유 슬롯이 채워질수록 원하는 결과를 얻기 어려워진다. 초반에는 대부분의 SSR 결과가 신규 획득으로 읽히지만, 수집 후반부에는 같은 결과가 반복되며 중복 SSR이 손실처럼 체감된다.

이 구간에서 유저는 다음과 같은 문제를 겪는다.

- 더 많은 재화를 사용해도 원하는 결과가 보장되지 않는다고 느낀다.
- 중복 SSR이 나오는 순간 소환 결과 전체가 실패처럼 읽힌다.
- 결과 확인 이후 이어지는 후속 행동이 없어서 뽑기 경험이 바로 종료된다.

이 시스템이 기대하는 변화는 다음과 같다.

- 중복 SSR을 즉시 손실이 아닌 후속 선택 기회로 전환한다.
- 수집 후반부에도 뽑기 지속 동기가 완전히 무너지지 않도록 한다.
- 소환 결과 확인 이후 `확정 관리`라는 후속 행동을 만든다.

## 3. 전체 플레이 흐름

이 시스템의 구현 기준 흐름은 아래 순서로 고정한다.

| 단계 | 유저 행동 | 시스템 처리 | 상태 변화 |
| --- | --- | --- | --- |
| 1 | 10회 소환 실행 | 보석 차감 후 결과 10개 생성 | `gemBalance` 감소, `lastSummonResults` 생성 |
| 2 | 결과 팝업 확인 | 결과를 화면에 표시 | 결과 팝업 오픈 |
| 3 | 결과 확인 | 결과를 즉시 보유 현황에 반영 | `ownedCounts` 증가 |
| 4 | SSR 결과 발생 시 | 해당 SSR을 확정 관리 대기 목록에 추가 | `pendingRows` 증가 |
| 5 | 결과 팝업에서 추가 소환 또는 닫기 선택 | 추가 소환 시 새 결과 생성, 닫기 시 메인 복귀 | 결과 팝업 유지 또는 종료 |
| 6 | 확정 관리 진입 | 대기 중인 SSR 목록 표시 | 확정 관리 팝업 오픈 |
| 7 | 재선택 실행 | 전환기 소모 후 후보 슬롯 개방 | `tokenBalance` 감소, `row.opened` 증가 |
| 8 | 최종 선택 | 원본 또는 후보 중 하나 확정 | 필요 시 `ownedCounts` 재조정 |
| 9 | 완료 확인 | 완료 팝업 확인 후 종료 | 선택 완료 행 제거 |

구현 시 중요한 점은 결과 팝업이 단순 확인창이 아니라는 점이다.  
결과는 팝업 노출 시점에 이미 보유 현황에 반영되어 있어야 하며, 확정 관리는 이 반영된 결과를 후속 정리하는 단계다.

## 4. 게임 전체 맥락에서의 역할

이 시스템은 10회 소환 결과 이후에 동작한다.  
중복 SSR 결과가 즉시 손실로 끝나지 않도록 후속 선택 행동을 만들고, 획득 결과를 보유 누적으로 이어주는 역할을 한다.

## 5. 화면 구성 및 구현 흐름

| 화면 | 표시 정보 | 입력 | 버튼 동작 | 상태 변화 |
| --- | --- | --- | --- | --- |
| 소환 화면 | 보유 보석, 확정 관리 진입, 영웅 인벤토리 진입, 10회 소환 버튼 | 10회 소환, 보석 추가, 확정 관리 진입, 영웅 진입 | 10회 소환 실행, 각 팝업 오픈 | 보석 차감, 결과 생성, 팝업 오픈 |
| 결과 팝업 | 최근 10회 결과, 보유 보석, 추가 10회 소환 버튼, 닫기 버튼 | 추가 소환, 보석 추가, 닫기 | `추가 소환` 클릭 시 팝업을 유지한 채 보석 차감 → 새 결과 10개 생성 → 보유량 즉시 반영 → SSR 대기 행 누적 → 결과 목록 갱신. `닫기` 클릭 시 메인 복귀 | `gemBalance`, `lastSummonResults`, `ownedCounts`, `pendingRows` 갱신 또는 팝업 종료 |
| 확정 관리 팝업 | 재선택 대상 목록, 전환기 보유량, 재선택 버튼, 최종 선택 버튼, 48시간 문구 | 후보 선택, 재선택, 전환기 추가, 최종 선택, 닫기 | `재선택` 클릭 시 전환기 차감 → 다음 후보 슬롯 개방 → 새 후보를 현재 선택값으로 지정. `최종 선택` 클릭 시 선택 대상 확정 → 필요 시 원본/후보 보유량 재조정 → 완료 팝업 표시 → 해당 대기 행 제거 | `tokenBalance`, `row.opened`, `row.selectedId`, `ownedCounts`, `pendingRows`, `completionState` 변경 |
| 영웅 보유 현황 팝업 | 보유 중인 영웅 목록, 보유 수량 | 스크롤, 닫기 | 목록 확인 후 종료 | 읽기 전용 |
| 완료 팝업 | 최종 선택된 영웅 정보 | 확인 | 팝업 종료 | `completionState` 초기화 |

구현 기준으로 보면, 개발자가 먼저 고정해야 할 것은 화면 수보다 화면 간 이동 순서다.  
소환 화면에서 결과 팝업으로, 결과 팝업에서 확정 관리로, 확정 관리에서 완료 팝업으로 이어지는 흐름이 끊기지 않아야 한다.

## 6. 핵심 규칙

| 규칙 | 내용 |
| --- | --- |
| SSR 유입 규칙 | 10회 소환 결과 중 SSR이 나오면 즉시 확정 관리 대기 목록에 추가한다. |
| 재선택 구조 | 원본 SSR은 유지하고, 후보 슬롯 2개를 순차적으로 연다. |
| 최종 선택 방식 | 원본 또는 후보 중 1개를 확정한다. |
| 결과 팝업 역할 | 결과 팝업은 확정 단계가 아니라, 이미 반영된 결과를 확인하고 연속 소환 여부를 결정하는 단계다. |

이 시스템의 핵심은 `중복 SSR을 그냥 두지 않는다`는 점이다.  
다만 보정 장치가 직접 획득보다 더 강한 가치로 읽히지 않도록, 원본 유지형 구조와 전환기 소모 규칙을 함께 둔다.

## 7. 데이터 및 재화 규칙

### 7-1. 데이터 상태

| 항목 | 역할 |
| --- | --- |
| `ownedCounts` | 영웅별 누적 보유량 |
| `pendingRows` | 확정 관리 대기 행 목록 |
| `lastSummonResults` | 최근 10회 소환 결과 |
| `gemBalance` | 보석 보유량 |
| `tokenBalance` | 전환기 보유량 |
| `completionState` | 완료 팝업 표시 대상 |

### 7-2. 재화 및 상태 변화 규칙

| 항목 | 증가 조건 | 감소 조건 | 표시 위치 | 비활성화 조건 |
| --- | --- | --- | --- | --- |
| 보석 | `+` 버튼 1회당 5000 증가 | 10회 소환 1회당 2000 감소 | 소환 화면 우측 상단, 결과 팝업 우측 상단 | 보석 2000 미만이면 10회 소환 버튼 비활성화 |
| 전환기 | `+` 버튼 1회당 10 증가 | 재선택 1회당 1 감소 | 확정 관리 팝업 헤더 | 전환기 0이면 재선택 버튼 비활성화 |
| 보유량 | 소환 결과 확인 시 즉시 증가 | 최종 선택 시 원본 외 대상 선택이면 원본 1 감소, 선택 대상 1 증가 | 영웅 보유 현황 | 없음 |
| 확정 관리 대기 목록 | SSR 결과 발생 시 행 추가 | 최종 선택 완료 시 행 제거 | 확정 관리 팝업 본문 | 목록이 비면 빈 상태 UI 표시 |

## 8. 예외 처리

| 상황 | 발생 조건 | UI 처리 | 상태 처리 |
| --- | --- | --- | --- |
| SSR 미획득 | 10회 결과에 SSR 없음 | 결과 팝업만 표시, 확정 관리 항목 없음 | `pendingRows` 변경 없음 |
| 확정 관리 비어 있음 | 대기 목록 0개 | 빈 상태 문구 표시 | 읽기 전용 |
| 보석 부족 | `gemBalance < 2000` | 메인/결과 팝업의 10회 소환 버튼 비활성화 | 소환 실행 차단 |
| 전환기 부족 | `tokenBalance = 0` | 재선택 버튼 비활성화 | 후보 개방 차단 |
| 미보유 영웅 획득 | `ownedCounts[target] = 0` 상태에서 획득 | `NEW` 배지 표시 | 획득 후 보유량 증가 |
| 보유 영웅 획득 | 기존 보유 중 획득 | `NEW` 배지 미표시 | 보유량만 증가 |
| 48시간 정책 | 현재는 정적 UX 문구만 제공 | `48시간 남음` 및 자동 발송 문구 표시 | 실제 자동 발송 로직 없음 |

구현 시 `48시간`은 기능이 아니라 UX 안내 문구라는 점을 명확히 유지한다.  
실제 만료 처리나 우편 발송 로직은 현재 범위에 포함하지 않는다.

## 9. 효율 검증 관점

이 시스템의 검증 포인트는 `원하는 SSR을 더 쉽게 준다`가 아니다.  
핵심은 중복 SSR 결과가 누적될수록 악화되는 손실 체감과 과금 효율 붕괴를 얼마나 완화하느냐에 있다.

효율은 아래 두 축으로 본다.

- 기대 비용 효율: 같은 목표를 달성하기 위한 평균 소환량과 평균 재화 소모가 얼마나 줄어드는가
- 체감 가치 효율: 중복 SSR이 즉시 손실이 아니라 후속 선택 기회로 읽히는가

특정 SSR 목표 접근성 개선은 이 시스템의 부차 효과다.  
본질은 중복 SSR이 “실패 결과”로 끝나지 않게 만드는 보정 구조를 제공하는 데 있다.

## 10. 리스크 및 성공 기준

### 주요 리스크

- 확정 관리 가치가 직접 소환 가치보다 더 강하게 인식될 수 있다.
- 대기 행이 과도하게 누적되면 관리 피로가 생길 수 있다.
- 재선택 체감이 약하면 손실 완화 장치로 기능하지 못한다.
- 결과 팝업의 연속 소환 편의성이 희귀 결과의 감정 가치를 약화시킬 수 있다.

### 성공 기준

- SSR 결과 발생 후 확정 관리 진입이 실제 후속 행동으로 이어진다.
- 재선택 완료율이 의미 있게 발생한다.
- 중복 SSR 획득 이후 즉시 이탈보다 후속 정리 행동이 늘어난다.

상세 계산 근거와 비교 가정은 내부 참고 문서인 [03_analysis.md](03_analysis.md)를 따른다.

## 11. 프로토타입 반영 범위 및 참고 기준

현재 프로토타입은 아래 범위까지 구현하고 검증한다.

- 10회 소환
- 결과 팝업
- 결과 즉시 보유량 반영
- SSR 확정 관리 유입
- 전환기 기반 재선택
- 영웅 보유 현황
- 정적 48시간 UX 문구

현재 문서는 실제 서비스 확정 수치 문서가 아니라, 프로토타입을 바탕으로 한 개발 기획서다.  
테스트용 확률, 초기 보유량, 임시 자산 상태, 세부 상태표는 내부 참고 문서인 [02_dev-spec.md](02_dev-spec.md)를 기준으로 확인한다.

참고 기준은 아래와 같다.

- 메인 구현 흐름 기준: 현재 프로토타입
- 세부 구현 및 테스트 기준: [02_dev-spec.md](02_dev-spec.md)
- 효율 해석 기준: [03_analysis.md](03_analysis.md)
