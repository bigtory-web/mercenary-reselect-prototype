# 소환 재선택 시스템 UI 레이아웃 규칙

이 문서는 `mercenary-reselect-prototype`에서 UI 수정 요청을 할 때 따르는 기준이다.
목적은 "예쁘게"가 아니라, 실제 게임 UX/UI 흐름에 맞는 화면 구조를 안정적으로 유지하는 것이다.

## 1. 기본 원칙

- 이 프로젝트의 UI는 웹페이지가 아니라 **게임 화면**처럼 보여야 한다.
- 구조를 먼저 확인하고, 그다음에 비율·배치·강조를 조정한다.
- 요청하지 않은 새 화면, 새 패널, 새 플로우는 임의로 추가하지 않는다.
- 레이아웃 변경은 코드 구조가 아니라 **실제 렌더 결과**를 기준으로 판단한다.

## 2. 화면 사용 범위

- 전체 기준은 **16:9 스테이지**다.
- 전체 화면 스크롤은 기본적으로 금지한다.
- 스크롤은 인벤토리처럼 필요한 경우에만 **내부 영역**에서 허용한다.
- 가능한 한 핵심 루프는 한 화면 안에 담는다.

## 3. 작업 전 확인 규칙

- 구조 변경이 필요하다고 판단되면 먼저 아래 2가지를 짧게 정리한다.
  - 현재 화면을 어떻게 이해했는지
  - 무엇을 왜 바꾸려는지
- 그다음에 수정한다.
- 특히 아래 항목은 먼저 확인 없이 추가하지 않는다.
  - 새 화면
  - 새 팝업 흐름
  - 좌측 리스트 / 우측 상세 패널
  - 별도 관리 단계
  - 불필요한 설명 블록

## 4. 게임 UI처럼 보이게 하는 기준

- 팝업은 실제 게임 팝업처럼 중앙에 떠야 한다.
- 배경 씬, 딤, 팝업 레이어 관계가 자연스러워야 한다.
- 정보 우선순위가 명확해야 한다.
- 같은 역할의 버튼은 크기, 무게감, 정렬을 최대한 통일한다.
- 단순 카드 전시보다 기능 보드/리스트처럼 읽히게 구성한다.

## 5. 텍스트 최소화 규칙

- 상태를 아이콘, 배지, 슬롯, 버튼 상태로 표현할 수 있으면 텍스트를 줄인다.
- 긴 설명 문단은 지양한다.
- 보조 설명이 꼭 필요하면 1줄 수준으로 제한한다.
- 상태 텍스트를 남길 때는 "정말 없으면 의미가 흐려지는가"를 먼저 확인한다.

## 6. 정보 밀도 규칙

- 큰 빈 박스나 의미 없는 여백을 만들지 않는다.
- 기능 UI는 예쁜 카드 전시보다 **빠르게 읽히는 구조**를 우선한다.
- 리스트형이면 한 행의 규칙이 반복되어야 한다.
- 같은 정보는 한 번만 보여주고, 반복 설명은 줄인다.

## 7. 소환 재선택 UI 규칙

- 재선택은 **리스트형 구조**를 기본으로 한다.
- 한 행은 한 재선택 대상이다.
- 각 행은 아래 요소로 구성한다.
  - 원본 슬롯
  - 후보 슬롯
  - 우측 액션 영역
- `좌측 목록 / 우측 상세` 구조로 바꾸지 않는다.
- 재선택 대상은 여러 개일 수 있다고 가정한다.
- 재선택 연출은 굴리는 방식보다 **빈 슬롯이 열리고 결과가 채워지는 방식**을 우선한다.
- 선택 상태는 한눈에 보이도록 외곽선, 글로우, 내부 강조로 명확히 구분한다.

## 8. 기본 상태 명확화

- 첫 화면은 소환 화면만 보여야 한다.
- 재선택 팝업은 기본적으로 닫혀 있어야 한다.
- 필요한 버튼을 눌렀을 때만 팝업이 열려야 한다.
- 결과 화면, 재선택 화면, 인벤토리 화면은 한 번에 하나의 메인 팝업만 열리게 유지한다.

## 9. 반복 UI 구현 규칙

- 같은 형태의 UI는 **공용 클래스 / 공용 레이아웃 규칙**으로 묶는다.
- 같은 목적의 버튼을 화면마다 따로 다르게 구현하지 않는다.
- 공용 규칙이 필요한데 개별 스타일이 늘어나면 먼저 공용화부터 검토한다.

## 10. 실패 방지 규칙

1. 레이아웃 요구사항은 코드 구조가 아니라 **실제 화면 결과 기준**으로 검증한다.
2. 같은 형태의 UI는 **공용 클래스 / 공용 규칙**으로 묶고, 개별 요소마다 따로 처리하지 않는다.
3. 구현 후에는 **브라우저에서 실제로 어떻게 보이는지 확인하기 전까지 완료로 판단하지 않는다.**

## 11. 수정 단위 운영 규칙

- 수정 포인트를 먼저 묶어서 정리한 뒤 한 번에 반영한다.
- 작은 수정마다 파일을 반복해서 변경하지 않는다.
- 확인도 항목별이 아니라 **묶음 기준**으로 진행한다.
- 수정 단위는 가능하면 아래처럼 묶는다.
  - UI 수정 묶음
  - 문서 수정 묶음
  - 기능 수정 묶음

## 12. 요청할 때 포함하면 좋은 정보

- 무엇을 바꾸고 싶은지
- 무엇은 유지해야 하는지
- 실제 게임처럼 어떤 감정을 원하고 있는지
- 스크린샷 기준 어느 위치가 문제인지

## 13. 금지사항

- 멋대로 구조를 갈아엎지 않는다.
- 요청하지 않은 새 단계를 추가하지 않는다.
- 기능보다 장식이 앞서는 레이아웃을 만들지 않는다.
- "보인다"가 아니라 "잘 읽힌다" 기준으로 판단한다.
- 화면 확인 없이 완료 처리하지 않는다.

## 14. 복붙용 요청문

```text
이 HTML 파일을 기준으로 UI를 수정해줘.

규칙:
- 먼저 현재 UI 문제를 3개 이하로 짚어줘.
- 내가 요청한 범위 안에서만 수정해줘.
- 새 화면, 새 패널, 새 플로우는 임의로 추가하지 말아줘.
- 16:9 전체 화면 기준과 내부 스크롤 규칙을 유지해줘.
- 같은 형태의 UI는 공용 레이아웃 규칙으로 맞춰줘.
- 코드 구조가 아니라 실제 화면 결과 기준으로 수정해줘.
- 수정 후 브라우저 기준으로 어떻게 보일지 확인하고 말해줘.
```
