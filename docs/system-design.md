# 소환 재선택 시스템

## 기획 의도

이 프로토타입은 `소환 결과를 먼저 체감한 뒤`, 그중 희귀한 SSR 결과만 다시 관리하는 흐름이 자연스러운지 검증하기 위한 시스템이다.

핵심 의도는 두 가지다.

- 10회 소환의 결과를 먼저 확인하게 해 뽑기의 감정 경험을 살린다.
- 중복되거나 아쉬운 SSR 결과는 `확정 관리`에서 다시 선택하게 해, 단순 획득이 아닌 후속 의사결정 경험을 만든다.

## 게임 전체 맥락에서의 역할

이 시스템은 독립적인 재화 소모 기능이 아니라, `소환 결과의 감정 경험을 후속 행동으로 연결하는 장치`다.

- 소환은 결과를 체감하는 메인 획득 경험을 담당한다.
- 확정 관리는 희귀 결과 중에서도 아쉬운 SSR 결과를 정리하는 후속 의사결정 경험을 담당한다.
- 영웅 인벤토리는 획득 결과가 실제로 누적되고 있다는 감각을 보완한다.

즉, 이 시스템의 역할은 SSR 중복을 단순 손실로 끝내지 않고, `소환 → 결과 확인 → 후속 선택`의 흐름으로 전환하는 데 있다.

## 도입 전 / 도입 후 유저 행동 변화

### 도입 전

- 유저는 10회 소환 결과를 확인한 뒤, 중복 SSR을 사실상 실패 경험으로 받아들일 가능성이 높다.
- 고등급 결과가 나와도 이미 보유한 용병이라면 감정적으로는 곧바로 종료 지점이 된다.
- 결과 확인 이후 추가 행동이 없기 때문에, 소환 경험이 단발성으로 끝난다.

### 도입 후

- 유저는 소환 결과를 먼저 체감한 뒤, SSR 결과가 `확정 관리`로 이어질 수 있다는 기대를 가진다.
- 중복 SSR은 즉시 실망으로 끝나지 않고, 후속 선택 기회로 전환된다.
- 결과 팝업, 확정 관리, 영웅 인벤토리가 연결되며 소환 이후의 행동이 한 단계 더 생긴다.

## 핵심 UX 의도

- 재선택은 메인 획득 수단이 아니라 `소환 결과를 정리하는 후속 보정 장치`처럼 느껴져야 한다.
- 유저는 먼저 SSR을 획득했다는 감정을 느껴야 하며, 이후에야 재선택의 가치를 판단해야 한다.
- 따라서 확정 관리는 기본 노출 기능이 아니라, `소환 결과를 본 뒤 의미가 생기는 기능`으로 설계한다.

## 시스템 구조

### 1. 소환

- 유저는 `10회 소환` 버튼으로 한 번에 10명의 결과를 확인한다.
- 결과는 별도 결과 팝업에서 먼저 보여준다.
- 소환 결과는 보유 현황에 즉시 반영되며, SSR 결과는 `확정 관리` 대기 리스트에 함께 추가된다.

### 2. 확정 관리

- 확정 관리는 미리 세팅된 상태로 시작하지 않는다.
- 10회 소환에서 SSR이 나와야만 리스트가 생성된다.
- SSR이 여러 개 나오면 각각 독립된 재선택 행으로 추가된다.

재선택 규칙:

- 원본 SSR은 유지된다.
- `재선택` 버튼을 누를 때마다 전환기 1개를 소모한다.
- 후보 슬롯은 순차적으로 열린다.
- 최종적으로 원본 또는 열린 후보 중 1개를 선택한다.

최종 선택 처리:

- 선택 완료 팝업을 보여준다.
- 해당 행은 확정 관리 리스트에서 즉시 제거된다.
- 원본과 다른 후보를 선택한 경우, 보유량도 최종 선택 기준으로 교체 반영한다.

### 3. 영웅 인벤토리

- 상단 `영웅` 버튼으로 진입한다.
- 지금까지 획득한 용병을 `중복 포함 보유량` 기준으로 보여준다.
- 정렬 기본값은 `등급 > 이름`이다.

### 4. 재화 구조

- `보석`은 소환을 반복하기 위한 메인 재화다.
- `전환기`는 재선택 시도를 제한하는 보조 재화다.
- 소환과 재선택이 서로 다른 재화를 사용하도록 분리해, 직접 획득 가치와 후속 보정 가치를 구분한다.

## 실패 케이스 및 운영 리스크

### 실패 케이스

- SSR 유입 빈도가 지나치게 높으면, 확정 관리가 보상 장치가 아니라 관리 피로 요소로 느껴질 수 있다.
- 재선택 효율이 직접 획득보다 높게 느껴지면, 소환 결과보다 후속 교환이 더 중요한 구조로 오해될 수 있다.
- 반대로 재선택 체감이 약하면 시스템이 존재해도 유저가 실질적인 의미를 느끼지 못할 수 있다.

### 운영 리스크

- `확정 관리 대기 상태`가 오래 누적되면, 유저는 기능을 구제 장치보다 미처리 보관함처럼 받아들일 수 있다.
- SSR 획득 만족감보다 재소환 편의성이 더 강하게 느껴지면, 결과 팝업의 감정 가치가 약해질 수 있다.
- 전환기 공급량이 과하면 재선택이 사실상 상시 재도전처럼 읽힐 위험이 있다.

## 검증 지표 및 성공 기준

### 핵심 검증 지표

- 10회 소환 이후 확정 관리 진입률
- SSR 결과 후 확정 관리 클릭률
- 재선택 사용률
- 최종 선택 완료율
- 재선택 진입 후 이탈률

### 성공 기준

- SSR 결과가 나왔을 때, 유저가 단순 종료하지 않고 후속 행동으로 이어질 것
- 확정 관리는 자주 열리되, 강제 노동처럼 느껴질 정도로 과도하게 누적되지 않을 것
- 재선택 사용은 발생하지만, 직접 획득보다 더 우선적인 가치로 인식되지 않을 것

## 효율 검증 관점

이 시스템은 단순히 `중복 SSR의 손실감을 완화하는 장치`에 그치지 않고, 특정 SSR을 목표로 소환하는 유저의 기대 효율을 바꾸는 구조다.

- 시스템이 없을 때 중복 SSR은 즉시 손실로 처리되며, 목표 SSR 획득 확률은 직접 소환 확률에만 의존한다.
- 시스템이 있을 때 중복 SSR은 `확정 관리`로 넘어가며, 후속 재선택을 통해 목표 SSR에 다시 접근할 수 있다.
- 따라서 동일한 소환 재화라도 `직접 획득 확률 + 후속 선택 기회`가 합쳐지며, 유저는 같은 과금으로 더 높은 목표 접근성을 기대할 수 있다.

이 효율은 긍정적 가치이지만, 동시에 주의할 점도 있다.

- 효율 상승폭이 지나치게 크면 직접 소환 가치보다 확정 관리 가치가 더 높게 느껴질 수 있다.
- 반대로 효율 상승폭이 너무 작으면 중복 SSR의 손실 체감을 충분히 완화하지 못한다.

즉, 이 시스템은 `소환 결과의 감정 정리`와 함께 `목표 SSR 획득 효율`을 어떻게 재구성하는지까지 검증해야 한다.

## 향후 확장 방향

- 확정 관리 대기 항목에 만료 UX를 도입해 방치감을 줄일 수 있다.
- 완전 선택형이 아닌 후보 제한형 재선택, 시즌형 재선택 재화 등으로 확장 가능하다.
- 영웅 인벤토리와 연계한 중복 성장 구조를 후속 시스템으로 붙일 수 있다.

## 프로토타입 반영 범위

- 이번 프로토타입은 소환, 결과 팝업, 확정 관리, 영웅 인벤토리의 흐름 검증을 우선한다.
- 등급별 구성 수, 확률, 초기 보유량, 시작 재화는 테스트용 세팅이며 시스템 본문 규칙으로 고정하지 않는다.
- SR/R 일부 용병은 임시 자산 및 임시 매핑을 사용한다.
- UI 자산(`manage-reselect.png`, `reroll-ticket.png`, `new-badge.png`)은 현재 흰 배경이 포함된 PNG이므로, 최종 품질 기준에선 투명 배경 버전으로 교체하는 것을 전제로 한다.
