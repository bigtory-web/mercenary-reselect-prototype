제출용 본문이 아니라, 실제 구현과 프로토타입 테스트를 함께 정리하는 작업용 메모로 사용한다.

## 1. 목적

- 화면별 UI 구성, 상태 전이, 규칙, 예외 처리를 한 문서에서 확인한다.
- 현재 프로토타입이 어떤 임시 수치와 자산으로 동작하는지 함께 기록한다.
- 개발자와 기획자가 동일한 테스트 기준을 공유할 수 있게 한다.

## 2. 현재 프로토타입 흐름

1. 첫 진입은 소환 화면이다.
2. `10회 소환` 실행 시 결과 팝업이 열린다.
3. 결과는 즉시 인벤토리에 반영된다.
4. 해당 10연에서 나온 모든 SSR은 확정 관리 대기 리스트에 추가된다.
5. 유저는 결과 팝업에서 닫거나 추가 10회 소환을 이어갈 수 있다.
6. 유저는 상단 `확정 관리` 버튼으로 들어가 SSR 재선택을 진행한다.
7. `영웅` 버튼으로 누적 획득 결과와 보유량을 확인한다.

## 6. 상태 전이

| 이벤트 | 입력 조건 | 처리 | 결과 상태 |
| --- | --- | --- | --- |
| 10회 소환 실행 | `gemBalance >= 2000` | 보석 2000 차감, 결과 10개 생성, 즉시 인벤토리 반영, SSR은 확정 관리에 추가 | `lastSummonResults`, `ownedCounts`, `pendingRows`, `gemBalance` 갱신 |
| 결과 팝업 닫기 | 결과 팝업 오픈 상태 | 팝업만 닫음 | 반영된 인벤토리/확정 관리 상태 유지 |
| 결과 팝업에서 10회 소환 | `gemBalance >= 2000` | 새 결과 10개 생성, 즉시 반영, SSR 추가 유입 | 기존 결과는 교체되지만 누적 상태는 유지 |
| 확정 관리 진입 | 확정 관리 버튼 클릭 | 리스트를 렌더링하며 팝업 오픈 | `pendingRows` 기준 표시 |
| 재선택 | `row.opened < 2` 그리고 `tokenBalance > 0` | 전환기 1 차감, 후보 슬롯 1개 개방, 새 후보 자동 선택 | `tokenBalance`, `row.opened`, `row.selectedId` 갱신 |
| 슬롯 직접 선택 | 열린 슬롯 클릭 | 해당 슬롯을 현재 선택 상태로 변경 | `row.selectedId` 갱신 |
| 최종 선택 | 행 존재 | 원본과 다른 경우 보유 수 조정, 완료 팝업 상태 생성, 해당 행 제거 | `ownedCounts`, `pendingRows`, `completionState` 갱신 |
| 완료 팝업 확인 | 완료 팝업 오픈 상태 | 완료 팝업 닫기 | `completionState = null` |
| 영웅 인벤토리 진입 | 영웅 버튼 클릭 | 인벤토리 렌더 후 팝업 오픈 | `ownedCounts` 기준 표시 |
| 보석 추가 | `+` 버튼 클릭 | 보석 5000 증가 | `gemBalance` 갱신 |
| 전환기 추가 | `+` 버튼 클릭 | 전환기 10 증가 | `tokenBalance` 갱신 |

## 7. 규칙 테이블

| 항목 | 규칙 |
| --- | --- |
| 소환 비용 | 10회 소환 1회당 보석 2000 소모 |
| 결과 반영 시점 | 소환 결과는 결과 팝업 표시 전에 즉시 인벤토리에 반영 |
| 확정 관리 유입 | 10회 소환 결과 중 SSR은 모두 즉시 확정 관리에 추가 |
| 재선택 비용 | 재선택 1회당 전환기 1 소모 |
| 후보 개방 수 | 원본 제외 후보 2개까지 순차 개방 |
| 최종 선택 | 원본 또는 개방된 후보 중 1개 선택 |
| 보유 수 반영 | 원본과 다른 후보 선택 시 원본 -1, 선택 후보 +1 |
| 보석 증가 | `+` 버튼 1회당 보석 5000 증가 |
| 전환기 증가 | `+` 버튼 1회당 전환기 10 증가 |
| 버튼 비활성화 | 보석 2000 미만이면 소환 버튼 비활성화, 전환기 0이면 재선택 버튼 비활성화 |

## 8. 예외 처리

| 상황 | 처리 정책 |
| --- | --- |
| SSR 미획득 | 확정 관리 목록 추가 없음 |
| 확정 관리 항목 없음 | 빈 상태 문구 표시 |
| 보석 부족 | 소환 화면/결과 팝업의 `10회 소환` 버튼 비활성화 |
| 전환기 부족 | `재선택` 버튼 비활성화 |
| 미보유 용병 획득 | `NEW` 배지 표시 |
| 보유 용병 획득 | `NEW` 배지 표시 안 함 |
| 48시간 경과 | 현재는 정적 UX 문구만 존재, 실제 자동 발송 기능은 미구현 |
| 영웅 인벤토리 항목 과다 | 팝업 내부 리스트만 스크롤 |
| 결과 팝업에서 연속 소환 | 이전 결과는 즉시 누적 반영된 상태로 유지, 새 결과로 교체 |

## 10. 프로토타입 테스트 세팅

- SSR / SR / R 총 24명 구성
- 테스트용 확률:
  - SSR 5%
  - SR 25%
  - R 70%
- 기본 보유 상태:
  - SSR 5명 보유
  - SR 8명 전원 보유
  - R 8명 전원 보유
- 보석 / 전환기 시작 수치는 흐름 검증용 임시값 사용

## 11. 검증 메모

- 10회 소환 결과를 본 뒤에만 `확정 관리` 필요성이 자연스럽게 느껴지는지 확인
- SSR만 재선택 대상으로 분리했을 때 UX 흐름이 이해 가능한지 검증
- `영웅 인벤토리`가 누적 획득 감각을 보완하는지 확인
- 재선택 시스템이 특정 SSR 접근성을 높이는 효과보다, 중복 SSR 손실 완화 체감에 더 기여하는지 확인

## 12. 초기 방향 메모

- 출발점은 `고등급 중복 결과가 감정적으로 손실처럼 느껴진다`는 문제의식이었다.
- 이전 방향은 교환이 직접 획득 가치보다 지나치게 강해질 위험이 있어 제외했다.
- 그래서 `보관 기반 후속 선택` 구조로 조정했다.
- 현재 방향은 `소환 결과를 먼저 체감한 뒤, 필요할 때만 후속 선택 경험으로 이어지는 구조`를 유지하는 것이다.

## 13. 자산 및 임시 구현 상태

- `manage-reselect.png`, `reroll-ticket.png`, `new-badge.png`는 원본 PNG 자체에 흰 배경이 포함되어 있어 추후 투명 배경 자산으로 교체 필요
- SR/R 일부 용병은 번호 기반 임시 매핑 자산 사용
