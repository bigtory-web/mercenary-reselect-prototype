# 소환 재선택 시스템

## 1. 시스템 개요

소환 재선택 시스템은 소환 결과 확인 이후, 중복 SSR을 후속 선택 경험으로 전환하는 보정 시스템이다.

이 시스템은 직접 획득 구조를 대체하지 않는다.  
대신, 중복 SSR이 즉시 손실로 끝나는 경험을 완화하고, 소환 결과 이후에도 유저가 한 번 더 의미 있는 의사결정을 할 수 있도록 설계한다.

핵심은 `더 쉽게 원하는 SSR을 주는 것`이 아니라, **운 요소로 인해 무너지는 과금 효율 체감을 완충하는 것**이다.

## 2. 문제 정의

수집형 가챠 시스템은 보유 슬롯이 채워질수록 원하는 결과를 얻기 어려워진다.  
초반에는 대부분의 고등급 결과가 가치 있게 받아들여지지만, 수집 후반부에는 이미 보유한 SSR이 늘어나며 중복 결과가 손실처럼 체감된다.

이 구간에서 유저는 다음과 같은 감각을 받기 쉽다.

- 원하는 결과를 얻기 위한 난이도가 급격히 상승한다.
- 더 많은 재화를 써도 성과가 보장되지 않는다는 인식이 강해진다.
- 중복 SSR이 쌓일수록 과금 가치보다 손실감이 더 크게 남는다.

정리하면 이 시스템이 해결하려는 문제는 다음과 같다.

1. 고등급 중복 결과가 손실처럼 체감되는 문제
2. 운 요소로 인해 과금 효율이 지나치게 나빠 보이는 문제
3. 소환 결과 확인 이후, 유저의 후속 행동이 끊겨버리는 문제

## 3. 게임 전체 맥락에서의 역할

본 시스템은 독립적인 교환 기능이 아니라, 메인 소환 루프의 감정 정리 장치다.

게임 전체 흐름에서의 역할은 다음과 같다.

- 소환: 메인 획득 경험 제공
- 결과 팝업: 희귀 결과 체감 및 가치 인지
- 확정 관리: 중복 SSR을 후속 선택 대상으로 전환
- 영웅 인벤토리: 획득 결과 누적과 보유 감각 보완

즉, 이 시스템은 `소환 → 결과 확인 → 후속 선택 → 보유 누적`의 구조를 만들고, 희귀 결과가 즉시 손실로 종료되는 빈도를 줄이는 역할을 한다.

## 4. 핵심 구조

### 소환

- 유저는 10회 소환을 통해 결과를 획득한다.
- 결과는 별도의 결과 팝업에서 먼저 보여준다.
- 결과는 팝업 확인 이전에 즉시 보유 현황에 반영된다.

### 확정 관리

- SSR 결과가 나왔을 때만 진입 가치가 생기는 후속 관리 기능이다.
- 중복 SSR은 확정 관리 리스트에 누적된다.
- 각 항목은 개별적으로 재선택 가능한 상태로 유지된다.

### 재선택

- 원본 SSR은 유지된다.
- 재선택 시 후보 슬롯이 순차적으로 열린다.
- 유저는 원본 또는 열린 후보 중 하나를 최종 선택한다.
- 재선택에는 전환기가 소모된다.

### 영웅 인벤토리

- 유저가 현재까지 획득한 용병과 보유량을 확인하는 화면이다.
- 소환과 재선택 결과가 실제로 누적되고 있다는 감각을 강화한다.

### 재화 구조

- 보석은 메인 소환 재화다.
- 전환기는 재선택 시도를 제한하는 보조 재화다.
- 두 재화를 분리해 직접 획득 가치와 후속 보정 가치를 구분한다.

## 5. 핵심 규칙

| 항목 | 규칙 |
| --- | --- |
| SSR 유입 | 소환 결과 중 SSR은 즉시 확정 관리 대기 리스트에 추가된다. |
| 결과 반영 시점 | 소환 결과는 결과 팝업 표시 전에 즉시 인벤토리에 반영된다. |
| 원본 유지 | 재선택 시 원본 SSR은 항상 유지된다. |
| 후보 구조 | 원본 외 후보 슬롯 2개까지 순차적으로 개방된다. |
| 재선택 비용 | 재선택 1회당 전환기 1개를 소모한다. |
| 최종 선택 | 원본 또는 열린 후보 중 1개를 선택한다. |
| 보유량 반영 | 원본 이외 후보 선택 시 원본 -1, 선택 대상 +1로 반영한다. |
| 만료 UX | 48시간 남음 문구를 통해 유예 기간 이후 자동 처리될 수 있음을 안내한다. |

## 6. 화면별 UX / 플로우

| 화면 | 핵심 요소 | 입력 | 출력 / 상태 변화 |
| --- | --- | --- | --- |
| 소환 화면 | 10회 소환 버튼, 보유 보석, 확정 관리, 영웅 | 10회 소환 / 확정 관리 클릭 / 영웅 클릭 / 보석 추가 | 소환 실행, 관련 팝업 진입, 보석 갱신 |
| 소환 결과 팝업 | 결과 카드 10장, 10회 소환, 닫기, 보유 보석 | 연속 소환 / 닫기 / 보석 추가 | 새 결과 생성, 팝업 종료, 보석 갱신 |
| 확정 관리 팝업 | 원본/후보 슬롯, 전환기, 48시간 남음, 재선택/최종 선택 | 슬롯 선택 / 재선택 / 최종 선택 / 전환기 추가 | 후보 개방, 선택 상태 변경, 완료 팝업 오픈, 행 제거 |
| 영웅 보유 현황 | 보유 카드 목록, 중복 수량, 내부 스크롤 | 스크롤 / 닫기 | 누적 보유량 확인 |
| 완료 팝업 | 최종 선택 결과, 확인 | 확인 | 완료 팝업 종료 |

기본 플로우는 다음 순서로 정리된다.

1. 10회 소환 실행
2. 결과 즉시 인벤토리 반영
3. SSR은 확정 관리 대기 리스트에 추가
4. 결과 팝업에서 결과 체감 또는 연속 소환
5. 확정 관리에서 재선택 진행
6. 최종 선택 후 보유량 재조정 및 항목 제거

## 7. 데이터 및 재화 규칙

### 핵심 상태값

| 데이터명 | 역할 |
| --- | --- |
| `ownedCounts` | 용병별 누적 보유 수량 |
| `pendingRows` | 확정 관리 대기 행 목록 |
| `lastSummonResults` | 최근 10회 소환 결과 |
| `gemBalance` | 보석 보유량 |
| `tokenBalance` | 전환기 보유량 |
| `completionState` | 완료 팝업 표시 대상 |

### 재화 규칙

| 항목 | 규칙 |
| --- | --- |
| 10회 소환 비용 | 보석 2000 소모 |
| 결과 팝업 재소환 | 보석 2000 소모 |
| 보석 증가 | `+` 버튼 1회당 5000 증가 |
| 전환기 증가 | `+` 버튼 1회당 10 증가 |
| 버튼 비활성화 | 보석 2000 미만이면 소환 비활성화, 전환기 0이면 재선택 비활성화 |

## 8. 예외 처리

| 상황 | 처리 정책 |
| --- | --- |
| SSR 미획득 | 확정 관리 목록 추가 없음 |
| 확정 관리 항목 없음 | 빈 상태 문구 표시 |
| 보석 부족 | 소환 화면/결과 팝업의 10회 소환 버튼 비활성화 |
| 전환기 부족 | 재선택 버튼 비활성화 |
| 미보유 용병 획득 | `NEW` 배지 표시 |
| 보유 용병 획득 | `NEW` 배지 미표시 |
| 48시간 경과 | 현재는 정적 UX 문구만 존재, 실제 자동 발송 기능은 미구현 |
| 영웅 인벤토리 항목 많음 | 팝업 내부 리스트만 스크롤 |
| 결과 팝업에서 연속 소환 | 이전 결과는 즉시 누적 반영된 상태를 유지하고 새 결과로 교체 |

## 9. 효율 검증 관점

본 시스템은 특정 SSR을 더 싸게 주는 시스템이 아니라, **운 요소로 인해 무너지는 과금 효율 체감을 완충하는 시스템**으로 해석해야 한다.

효율 검증은 두 축으로 본다.

1. 기대 비용 효율  
   특정 SSR 1종을 목표로 할 때 필요한 평균 소환 비용이 얼마나 줄어드는가
2. 체감 가치 효율  
   중복 SSR이 즉시 손실이 아니라 후속 선택 기회로 받아들여지는가

핵심은 효율 상승 자체가 아니라, **소환을 계속할 이유가 완전히 사라지지 않도록 손실 체감을 완충하는가**에 있다.

## 10. 리스크 및 성공 기준

### 주요 리스크

- SSR 유입 빈도가 지나치게 높으면 확정 관리가 보상 장치보다 관리 피로 요소로 느껴질 수 있다.
- 재선택 효율이 직접 획득보다 높게 느껴지면, 유저가 결과 팝업보다 후속 정리 기능을 더 중요한 가치로 인식할 수 있다.
- 반대로 재선택 체감이 약하면 손실 완화 장치로 기능하지 못한다.
- 대기 상태가 오래 누적되면 구제 장치가 아니라 미처리 보관함처럼 느껴질 수 있다.
- SSR 획득 만족감보다 연속 소환 편의성이 더 강해지면 희귀 결과의 감정 가치가 약해질 수 있다.

### 주요 검증 지표

- 10회 소환 이후 확정 관리 진입률
- SSR 결과 후 확정 관리 클릭률
- 재선택 사용률
- 최종 선택 완료율
- 재선택 진입 후 이탈률

### 성공 기준

- SSR 결과가 나왔을 때 유저가 단순 종료하지 않고 후속 행동으로 이어질 것
- 확정 관리가 자주 열리되, 강제 노동처럼 느껴질 정도로 과도하게 누적되지 않을 것
- 재선택 사용은 발생하지만, 직접 획득보다 우선적인 가치로 인식되지 않을 것

## 11. 프로토타입 반영 범위

- 이번 프로토타입은 소환, 결과 팝업, 확정 관리, 영웅 인벤토리의 흐름 검증을 우선한다.
- 등급별 구성 수, 확률, 초기 보유량, 시작 재화는 테스트용 세팅이며 시스템 본문 규칙으로 고정하지 않는다.
- 일부 용병 자산은 임시 매핑을 사용한다.
- 일부 UI 자산은 최종 품질 기준에서 교체가 필요한 상태다.
