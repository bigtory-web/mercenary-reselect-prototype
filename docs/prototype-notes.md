# mercenary-reselect prototype notes

## 목표

- 10회 소환 결과를 본 뒤에야 `확정 관리`의 필요성이 자연스럽게 느껴지는지 확인
- SSR만 재선택 대상으로 분리했을 때 UX 흐름이 이해 가능한지 검증
- `영웅 인벤토리`가 누적 획득 경험을 보여주는 보조 화면으로 동작하는지 확인

## 이번 프로토타입의 핵심 흐름

1. 첫 진입은 소환 화면이다.
2. `10회 소환`을 누르면 결과 팝업이 열린다.
3. 결과는 즉시 보유 현황에 반영되며, 해당 10연에서 나온 모든 SSR이 `확정 관리` 대상이 된다.
4. 유저는 상단 `확정 관리` 버튼으로 들어가 재선택을 진행한다.
5. `영웅` 버튼으로 누적 획득 용병과 보유량을 확인할 수 있다.

## UX 의도

- 재선택은 독립 메뉴가 아니라 `소환 결과를 본 뒤의 후속 행동`처럼 느껴져야 한다.
- 확정 관리는 “SSR이 실제로 나왔을 때만 의미가 생기는 기능”으로 보이게 한다.
- 영웅 인벤토리는 뽑기 결과가 실제로 축적되고 있다는 감각을 보완한다.

## 레이아웃 규칙

- 전체 화면은 16:9 스테이지 고정
- 전체 화면 자체는 스크롤하지 않음
- 스크롤이 필요한 경우는 인벤토리나 리스트처럼 내부 영역만 허용
- 팝업은 중앙 오버레이 구조로 처리

## 프로토타입 테스트 세팅

- SSR, SR, R 총 24명 구성
- SSR 확률 5 / SR 25 / R 70
- 기본 보유 상태
  - SSR 5명 보유
  - SR 8명 전원 보유
  - R 8명 전원 보유
- 보석 시작 수치와 전환기 시작 수치는 UX 테스트용 예시값을 사용

## 구현 메모

- 확정 관리 리스트는 초기 비어 있음
- 재선택 시 전환기 1개 차감
- 완료 팝업은 최종 선택 후 확인 버튼으로 닫음
- 결과 팝업은 이미 반영된 결과를 확인하고 연속 소환을 이어가기 위한 화면으로 사용

## 효율 검증 가정

- 목표는 `특정 SSR 1종 획득`으로 고정한다.
- 비교 시나리오는 두 가지다.
  - 시스템 없음: 일반 10회 소환만 반복
  - 시스템 있음: 10회 소환 결과 중 SSR은 모두 확정 관리로 유입
- 현재 프로토타입 기준 재선택 구조는 `원본 + 후보 2개`다.
- 후보 2개는 원본을 제외한 SSR 풀에서 중복 없이 생성한다.
- 전환기 차감은 효율 계산에 포함하되, 이번 비교에서는 `전환기를 사용할 수 있는 상태`를 기본 가정으로 둔다.

## 확인하고 싶은 검증 포인트

- 유저가 SSR 결과를 먼저 긍정적으로 체감한 뒤, 재선택을 후속 선택지로 받아들이는지
- 확정 관리가 메인 메뉴보다 소환 결과의 자연스러운 연장선처럼 느껴지는지
- 영웅 인벤토리가 누적 획득 감각을 실제로 보완하는지
- 특정 SSR 1종을 목표로 할 때, 확정 관리 시스템이 기대 소환 비용을 얼마나 줄이는지
- 중복 SSR이 즉시 손실이 아니라 후속 선택 기회로 받아들여지는지

## 자산 및 임시 구현 상태

- `manage-reselect.png`, `reroll-ticket.png`, `new-badge.png`는 현재 원본 PNG 자체에 흰 배경이 포함되어 있어 추후 투명 배경 자산으로 교체 필요
- SR/R 일부 용병은 번호순 임시 매핑 자산을 사용
- 일부 수치와 보유 상태는 밸런스 결정보다 흐름 검증을 위한 테스트용 세팅
